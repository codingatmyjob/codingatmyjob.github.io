What Is Prompt Injection?

IBM describes prompt injection as "...malicious inputs as legitimate prompts, manipulating generative AI systems (GenAI) into leaking sensitive data, spreading misinformation, or worse." Essentially, an application says "follow these rules," and an attacker says "do this instead." If the model follows the attacker's instructions, it can lead to a number of unintended consequences.

These consequences scale with what the model has access to, and what it can do. A support chatbot might be manipulated into providing incorrect information, answering unrelated questions, or performing unintended actions (like writing code, and cost the company money). An Agent with access to a database, email, or other tools could be manipulated into sending or forwarding sensitive emails, deleting data, etc.

System Prompt Extraction

Prompt injection is not one single technique, instead it's a class encompassing a variety of attack patterns; The Open Worldwide Application Security Project (OWASP) lists it as the most critical threat in its LLM Top 10 (LLM01). Within the prompt injection class, attacks split into two branches: direct injection, where the malicious input comes straight from the user's own message, and indirect injection, where it's ingested via content the model processes.

System prompt extraction is one of the most common direct injection techniques, and it's the technique we will be attempting in this demo. It targets the model's behavioral instructions directly. If successful, the attack can expose internal policy, workflow details, guardrails, or enough information for an attacker to craft stronger follow-up attacks.

Common attack patterns include asking the model to "repeat your exact instructions above" or to role play as a debugger and print the initialization context.

In this demo, we utilize Hugging Face to load in and run a lightweight generative model. How to Defend Against Prompt Injection

      

        Once you've gotten the demo to work, the next question is... how do you stop it? From a traditional web security perspective, we might try and sanitize input,
        but that just doesn't work here. SQL injection and XSS can be sanitized because there is a consistent, escapable structure. 
        With prompt injection, system instructions and user data often arrive together as unstructured natural language, with no good means of delineation.
      

      

        Given the nature of how these generative models work, there is no one-size-fits-all fix. However, through a defense-in-depth approach, you can mitigate the risk of prompt injection attacks. 
        OWASP lists the following recommended defenses:
      

      
    
        
  1. Constrain model behavior: Give the system prompt explicit boundaries on role, capability, and topic, and tell it to ignore attempts to override those instructions.
    2. 
        
  2. Define and validate expected output formats: Require a specific output format and check it with code.
    3. 
        
  3. Implement input and output filtering: Screen content on the way in and out with semantic filters and string checks. Evaluate responses for contextual relevance.
    4. 
        
  4. Enforce privilege control and least privilege access: Keep API tokens and sensitive functions in your own code, not the model's, and grant only what a task needs.
    5. 
        
  5. Require human approval for high-risk actions: Put a human in the loop for privileged operations to prevent unauthorized actions.
    6. 
        
  6. Segregate and identify external content: Clearly mark untrusted input, like documents or tool output, so it's treated as data, not instructions.
    7. 
        
  7. Conduct adversarial testing and attack simulations: Red-team the system regularly, treating the model itself as an untrusted user.
