Overview

This is a simple LLM enabled chatbot demo that connects a Groq-hosted LLM to the NOAA tide station data via API. Ask about tide predictions or current water levels at any U.S. coastal station. The goal was to build an interactive frontend similar to the Typeahead demo, but have an actual backend that handles LLM tool calling.

Hey! Ask me about tides anywhere along the U.S. coast — I can look up predictions, high/low times, and current water levels using real NOAA data.

Rate limited to 10 requests per hour per IP. Powered by Groq + NOAA CO-OPS API.

The Token Problem

I can't expose an API key (like Copilot) in a public demo for obvious reasons. The solution was a free-tier LLM to demo with, and I ended up choosing llama3-70b-8192 on Groq. The model has plenty of free calls, and I easily configured simple controls and abuse prevention; IP rate limiting (10/hour/IP), Request size limit (inputs over 500 characters rejected), and a CORS whitelist for only this domain.

Architecture

GitHub Pages hosts a static frontend with JS React and Vite. It can't execute server-side code, so a separate Node.js backend on Fly.io handles the LLM calls, enforces rate limits, and keeps secrets hidden.

When the user sends a message, Groq decides which tool to call but can't reach NOAA directly. It returns a specific instruction to the backend indicating the tool to use. The backend then fetches the data and sends it back to Groq to generate the final response which is streamed back to the user on the frontend.

  Browser (GitHub Pages)
          ↓ POST /chat
  Fly.io — Node.js backend
          ↓ message
  Groq (LLaMA 3)
          ↓ "call get_tides(station, date)"
  Fly.io — Node.js backend
          ↓ fetch
  NOAA API
          ↓ data
  Fly.io — Node.js backend
          ↓ data + original message
  Groq (LLaMA 3)
          ↓ streamed response
  Browser (GitHub Pages)

LLM Powered Chatbot vs MCP

Model Context Protocol (MCP) servers dynamically expose tools, resources, and prompts at runtime, so a client can discover what's available without anything being hardcoded. Traditional LLM tool-calling requires every tool to be defined upfront and shipped with the application. MCP also operates over a standardized protocol, meaning any compatible client can connect to any MCP server with no custom integration required. MCP clearly has an edge in flexibility, but the runtime loops of MCP and traditional LLM chatbots look identical, and both approaches share more than they don't.

They are incredibly similar, in fact, BOTH:

  • Rely on the LLM to determine which tools to call
  • Require specific tool definitions (name, parameters, description)
  • Follow the same pattern: LLM → pick tool → backend processes task → return result to LLM
  • Can use the same tool types (fetch, database queries, etc.)
  • Are only as good as the LLM doing the processing

My biggest complaint with MCP is that it doesn't really feel new. It's an old dog with new tricks. The crucial difference between the two lies in the additional resources and reasoning capabilities MCP clients can leverage at run time. With MCP, models can create incredibly unique workflows on the fly, chaining tool calls together to achieve complex results. With a traditional LLM chatbot, the tools and workflows are more or less fixed.
