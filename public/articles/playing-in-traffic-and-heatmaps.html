Playing in Traffic and Heatmaps

May 02, 2026
Project JavaScript WebApp API

I Like Hexagons

Inspired by the Google Maps API, I thought it would be really cool to try and create a hex oriented activity heatmap (with a last second traffic congestion overlay). From what I could find, the Maps API didn't have anything readily available to accomplish this task, so instead I simulated congestion by generating a per-hex hourly activity curve using commute-shaped Gaussian peaks (morning/evening), overnight dips, and included some randomness so each hex was unique. Then I factored in a road-density baseline from OpenStreetMap to make it look plausible.

After playing around with this, trying to tweak it and make it look legitimate (why is a neighborhood more active than the busiest intersection in town??), I did some research (probably should have started with this) and found that not only does TomTom still exist, but they have an awesome API that provides aggregated historical traffic data at a per-street level. After finally determining the scope of my project I finalized the requirements: A semi-accurate hexagon-based activity heatmap, with a historical traffic overlay, and a slider at the bottom to visualize the time of day... with a few other scope creep inclusions like per-hex tooltips, ZIP code search, etc.

Below is the documentation of my personal experience with each API and library I used. I documented the cost of each API/library (we're broke out here), the ease of use, and finally a brief explanation of how it was used or integrated into the overall project.

What the App Does

The app renders a live Google Maps basemap overlaid with an Uber H3 hexagonal grid. Each hex is colored on a gradient from cool blue (low activity) to hot red (high congestion) based on a traffic profile unique to that hex's location and road density. On top of the hex layer, TomTom road segments are drawn in orange and red wherever historical speed data shows significant congestion.

The four main features are:

  • Hex grid: The entire visible viewport is tiled with H3 resolution-9 hexagons. Each cell is colored based on its traffic profile at the given hour. The color gradient dynamically normalizes to the min/max of the visible cells, so the full color range always spans whatever is on screen.
  • TomTom road overlay: Orange and red polylines are drawn over real road segments fetched from TomTom's historical Flow Statistics API. Only segments with a normalized congestion score ≥ 0.6 (orange–red range) are shown.
  • ZIP code search: Entering any US ZIP code recenters the map and rebuilds the hex grid at that location. The Google Geocoding API resolves coordinates and a place label (e.g. "Manhattan, NY") is shown as feedback below the search field.
  • Per-hex tooltip: Clicking any hex opens a popup showing the current traffic level, peak hour, quietest hour, and a 24-bar daily sparkline profile. The active hour is highlighted in the sparkline as the time slider moves.

The demo was created capturing static images with the slider hidden on the actual webapp, injecting those images here, and then reattaching the slider. The time slider at the bottom advances the hour from 00:00 to 23:00. Every hex and road segment re-evaluates its color at the selected hour, with smooth transitions between frames. The Fake Traffic Problem

Most of what the time slider shows for the hex grid is synthesized. It resembles true commute patterns using Gaussian morning/evening peaks, overnight dips, and per-hex noise. Hexes near road corridors got a slight boost in activity score, and hexes far from any road stayed low. It felt spatially plausible, but still a mathematical approximation.

Per-street historical traffic data didn't exist for free at the scale I needed. TomTom's statistics endpoint has a daily limit of 2,500 daily requests, Google doesn't expose anything comparable. The hexes that show TomTom segments contain real data; everything else is an approximation based on the hex's road density and a synthetic traffic curve. I blended the TomTom congestion score on top of the baseline to get a final congestion score for each hex, which is what drove the color.

Google APIs

All Google Maps Platform APIs share a $200/month free credit. For personal projects this effectively makes most Google APIs free, though billing must be enabled on your Cloud project to generate a key at all.

Google's Maps JavaScript API

Cost: $7 per 1,000 map loads

Ease of use: Easy

How it was used: Google Maps provided the basemap and camera controls, loaded in with @googlemaps/js-api-loader. I originally wanted a tilted basemap with raised 3D hex columns reflecting activity levels. My first attempt was deck.gl's HexagonLayer, but that approach was not a good fit, because I needed stable, globally addressable hex cells for caching and per-hex state. I dropped the raised-column idea and moved to H3-indexed hexes, rendered with deck.gl PolygonLayer, which gave me predictable geometry and global IDs.

My second issue was tilt. Google Maps defaults to raster rendering, which is essentially a flat map layer with no available 3D camera behavior. To enable tilt, I had to generate a Map ID with vector rendering enabled. I integrated everything correctly, went to test my code, but my map in Chrome was still flat. After about 15 minutes of debugging and testing, I switched browsers. To my surprise, the tilt worked great in Edge and the VS Code browser. The issue was something silly; hardware acceleration was disabled in Chrome, which caused a silent default to raster mode. Apparently tilting the map is a GPU intensive task, who knew.

Google's Geocoding API

Cost: $5 per 1,000 requests

Ease of use: Easy

How it was used: I used this to power the ZIP code search in the top-left panel. You typed a US ZIP, hit Go, and the Geocoding API returned the coordinates and a formatted address. The app pulled geometry.location to recenter the map and read the address components to build a label like "Manhattan, NY" shown as feedback below the search field. It was straightforward, no issues.

Google's Places API

Cost: Nearby Search $0.032/request, Place Details billed per field: Basic $0.017, Contact $0.03, Atmosphere $0.05

Ease of use: Easy

How it was used: I tried to use Places to populate a "Nearby Places" section in my hex tooltips that would include restaurants, transit stops, anything that might explain why an area was busy. The problem was that Nearby Search frequently returned street names (routes) or town names (localities) instead of actual businesses, which was useless for contextualizing traffic. When it did return businesses, it felt random and inconsistent, and these were not the businesses causing traffic (Subway, McDonald's, Laundromat, etc.). I stopped using it after a few test cycles.

TomTom APIs

Yes, TomTom is still around, and they have a robust collection of APIs from decades of aggregated traffic data. The free tier includes 50,000 tile requests/day and 2,500 non-tile requests/day (separate limits). It was my only source of real traffic data for this project. One cool thing about TomTom's API is you don't need to upload a credit card when you sign up, the API will simply start to block requests when you hit your daily cap.

TomTom's Traffic Tile API

Cost: Included in the 50,000/day free tier

Ease of use: Easy

How it was used: I tried this first since it felt like a lightweight solution. I encountered an immediate hard stop: tiles are pre-rendered images fetched by zoom/x/y grid coordinate. There was no underlying data to query; I couldn't ask what the speed was on a given road, click a segment, or map time-of-day values to the time slider. I stopped using this very quickly.

TomTom's Flow Segment Data - Real-Time

Cost: Included in the 2,500/day free tier

Ease of use: Easy

How it was used: I tried flowSegmentData/absolute as an initial data source just to see how it worked. By passing it a lat/lng and zoom level, it returned the nearest road segment's current speed, free-flow speed, and polyline geometry. As this was strictly real-time traffic, it couldn't be used with the time slider.

TomTom's Flow Segment Data - Statistics (Historical)

Cost: Included in the 2,500/day free tier

Ease of use: Medium*

How it was used: This was the endpoint that actually integrated well with the time slider. Using flowSegmentData/statistics, I got a sub-hourly historical speed profile across the full day. As I quickly found out, at street-level granularity across a zoomed-out viewport, I burned through it in minutes.

Coverage problem: TomTom's historical data comes from GPS probes. These are cars that drive around and report their speed and location. The more traffic a road sees, the more probe data exists for it. Major roads in big cities were well covered; quiet residential streets and anything outside major metro areas had little to no data at all.

Requests limitation: This was why I marked it Medium*. The API was straightforward, but I had to do mental gymnastics to stay within my request budget. TomTom's free tier is 2,500 requests per day, and a city viewport at zoom 15 could have 200+ hexes with one hit each, per slider step. Panning around Manhattan with the slider playing burned through the daily cap in minutes. I attacked this by only requesting data for hexes above a certain OSM road density score, and only once per whole hour rather than on every fractional step.

While this seriously limited my traffic visibility, it eliminated the majority of idle API usage. It also allowed me to continue testing the API without hitting the daily cap in 4 minutes when my cap reset (I still only lasted about 20 minutes).

OpenStreetMap - Overpass API

Cost: Free

Ease of use: Medium

How it was used: Overpass let me query what roads actually existed near each hex center, and I used that to build a baseline score for each cell to try and represent potential activity. For example, a hex sitting on top of a highway got a higher score than one sitting in a park. When TomTom data eventually came in for sampled points, it was blended on top of those baseline scores, and the result drove both the hex color and whether a hex was worth spending a TomTom request on in the first place.

The limitation was that Overpass only knew about roads, not what was generating traffic on them. A crazy example of this phenomena is a neighborhood. Neighborhoods can have very complex street grids, and OSM will interpret these as higher road density (busier), when they're actually quiet residential streets, vice 4 way intersections, where it only sees 2 multilane roads.

H3 - Uber's Hexagonal Grid

Cost: Open-source library (free)

Ease of use: Easy

How it was used: H3 became the core hexagonal grid system for the app. I used it to fill the visible map area with hex indexes at a fixed resolution, which gave me a deterministic set of cells every time the camera moved.

With a stable grid in place, I drew each hex cleanly on the map and used each one as a reference point to collect specific traffic data. Any location I clicked or searched for could be matched to a specific hex, which made cell-specific API calls very easy. Now instead of constantly recalculating everything from scratch, I had the ability to cache data to specific hexes for reuse.

If I Could Do Something Differently

I went about fulfilling my requirements wrong, and backwards. I had initially tried to establish an activity heatmap with synthesized data, and then later decided to layer historical traffic data (what I didn't realize was basically activity) on top. In retrospect, the historical traffic data TomTom provided was the closest thing to an activity heatmap I could use as a starting point for free.

I should have started with the TomTom historical data at a small scale, then layered my hexagons and done the heatmap calculations based on traffic, rather than the other way around. This would have provided a consistent baseline across all types of areas and infrastructure. Hex grid coloring could have easily coincided with real traffic congestion patterns, and resembled a real activity heatmap instead of an approximation.
